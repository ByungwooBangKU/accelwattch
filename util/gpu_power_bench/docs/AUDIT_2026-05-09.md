## HIGH severity

### H1. H100 PCIe / NVL 변종이 silently SXM 으로 분류됨 — **WONTFIX (out-of-scope)**

> **Status (2026-05-09 review):** 사용자 결정으로 H100 PCIe / NVL 지원은
> 작업 범위 밖. 본 finding 은 record-keeping 목적으로만 유지. 대상 하드웨어
> (`rtx3090` / `a100_sxm` / `h100_sxm`) 외 변종을 사용하는 외부 사용자에게만
> 영향이 있음.

`gpu_profiles.py:103` 의 `infer_gpu_profile()`:

```python
if "h100" in hay or "hopper" in hay or "gh100" in hay:
    return "h100_sxm"
```

`h100_pcie` 또는 `h100_nvl` profile 정의 자체가 없음. NVIDIA 공식 H100 페이지
(<https://www.nvidia.com/en-us/data-center/h100/>) 의 spec 표는 SXM 과 NVL
(PCIe form-factor) 의 spec 을 별도 컬럼으로 둠 — memory BW / TDP / FP8
Tensor Core throughput 이 둘 사이에 다름. 외부 사용자가 H100 NVL (PCIe
form) 보드를 가지고 본 suite 를 돌리면 동일하게 `h100_sxm` 으로 라우팅됨.
compute capability 는 둘 다 `9.0` 이라 `cc` fallback 조차 분기 안 됨.

`resolve_gpu_profile()` 가 mismatch 시 `reason` 만 채우고 block 하지 않음
(`gpu_profiles.py:128-129`) — `--gpu-profile auto` 면 warning 도 안 뜸 (관측
profile 이 `h100_sxm` 과 동일하게 잡히기 때문). HBM3 pJ/bit headline 이 SXM
가정으로 plot → literature 비교 reference 가 잘못됨.

본 프로젝트의 headline 대상 GPU 가 SXM 변종으로 한정되어 있어 fix 우선순위
없음. 향후 PCIe / NVL 측정이 필요해지면 `gpu_profiles.py` 에 별도 profile
추가 + `infer_gpu_profile()` 의 product-name 분기.

## MEDIUM severity

### M1. TestCases.md A.1 의 `--cache-sweep` 셀 수가 outdated

`TestCases.md:70`:

> `--cache-sweep` 사용 시엔 11 load 가 3 regime 점 (L2-resident / partial /
> dram-stream) 으로 줄어 5 × 2 × **3** = 30 cell

실제 `benchmarks.py:451-452`:

```python
def cache_sweep_points(...) -> list[int]:
    """Return 5 N values that each land in the centre of one cache regime."""
```

3-bucket → 5-bucket 으로 업그레이드된 시점에 TestCases.md 가 안 바뀜. 실제
cache-sweep cell 수 = `5 × 2 × 5 = 50` (문서 30 보다 67 % 많음). 사용자가
sweep wall-time 잘못 추정.

**권장 fix**: TestCases.md A.1 의 cell-count 식과 `--cache-sweep` flag
설명을 5-regime 으로 갱신.

### M2. REVIEW.md Axis 3 요약이 L2 probe 추가 후 outdated

`REVIEW.md:11`, `:16`, `:131` (3 곳):

> `3. Memory hierarchy 분리 | △ | DRAM read/write/marginal 깨끗. **L2 미만
> (L1/SMEM/register) 은 bundled**`
>
> **한 줄 평** — DRAM 까지의 component 분리는 rigorous. **L2 안쪽 (L1 / register
> / pure compute) 은 NVML 측정의 *근본적 한계* 라 bundled** …

### M3. L2 auto-R sizing 의 `k_guess_pj_bit` default 가 실측보다 클 가능성

`gpu_power_bench.py:765` (CLI default) + `:342-347` (auto sizing):

```python
k_j_per_bit = max(k_guess_pj_bit, 1e-6) * 1e-12   # default 1.0 pJ/bit
bits = max(1, int(window_bytes) * 8)
base = ceil(target_energy_j / (k_j_per_bit * bits))
# auto sweep returns sorted({R/4, R/2, R, 2R})
```

`--l2-k-guess-pj-bit` 기본값 1.0 pJ/bit 으로 `base` 가 산정되고, auto sweep
은 그 R 의 `{R/4, R/2, R, 2R}` 4 점.

**우려 (실측 근거 필요)**: 실제 H100 L2-hit traffic path pJ/bit 가 default
1.0 보다 작으면 — 일부 보고에서 Hopper L2 efficiency 가 sub-pJ/bit 영역으로
인용되지만 본 suite 의 실측치는 아직 commit 되지 않은 상태 — center point R
이 target 10 J 에 도달하지 못함. 4 점 sweep 의 max (`2R`) 가 ~5-7 J 수준에
머무는 시나리오 발생 가능. 이 경우 NVML 적분 windows 가 노이즈 floor 에 더
노출됨.

다만 **실측 기준 (e.g., 첫 H100 run 의 `_02_l2_summary.csv` 의 pJ/bit
headline) 이 commit 되기 전엔 specific 한 numeric mismatch 라기보다 잠재적
under-shoot 임**. 실제 fix priority 는 첫 H100 run 이후 결정하는 게 맞음.

**권장 fix (실측 후)**: 첫 H100 / A100 / RTX 3090 run 의 measured pJ/bit
headline 을 base 로 profile 별 `k_guess_pj_bit` default 등록, 또는 첫 패스
결과로 second-pass auto-correction.

### M4. `--cache-sweep` 모드의 per-regime slope 가 항상 1-point fallback

`benchmarks.py:451`: cache-sweep 은 regime 마다 정확히 1 N 값 만 잡음.
`summarize_by_regime` 의 WLS fit 이 (op, dtype, regime) 단위라 → 모든
cache-sweep 셀이 `n_points=1`, slope = `y/x` (per-cell J/element 그 자체).

이 자체는 의도된 설계지만 부작용:

- bootstrap CI 가 NaN (n=1 → resample 의미 없음).
- 사용자가 `slope_dyn_ci_lo/hi` 를 보고 "regression confidence" 로 오해 가능.
- gelu 노이즈 floor 이슈 (median fallback 으로 fix 한 것) 의 root cause 이기도.

**권장 fix**: `cache_sweep_points` 가 regime 당 2 개씩 N 값을 잡도록 변경
(5 → 10 개) 하면 regime-내부 slope 가능. 측정 시간은 2× 증가.

### L4. `headline_status` 의 row-level 3-level vs report-level 6-level 표기 차이 (docs clarification)

`gpu_profiles.headline_status()` 는 row-level 에 `HEADLINE / PROXY /
NOT_HEADLINE` 3-level 을 stamp. `component_validation_report.py:400-419` 가
report 단계에서 R² / coefficient quality 로 6-level
(`pass / low_conf / not_headline / not_applicable / missing / fail`) 도출.

REQUIREMENTS.md §11 의 정확한 문구는 "**Report classification** must
distinguish: pass / low_conf / not_headline / not_applicable / missing /
fail" — 즉 *report* 단계 의무이지 row-level 컬럼 의무가 아님. 따라서 코드와
spec 자체는 일치하나, "row 단위 stamp 와 report 단위 classification 이
서로 다른 어휘" 를 사용자가 같은 것으로 오인할 가능성이 docs clarification
수준의 risk.

**권장 fix (docs only)**: REQUIREMENTS.md §11 또는 README 의 헤드라인
컬럼 설명에 "row stamp = HEADLINE / PROXY / NOT_HEADLINE (3-level), report
classification = pass / low_conf / not_headline / not_applicable / missing
/ fail (6-level, derived from row stamp + R²/CI quality gates)" 명시.
